# Language Semantics

Этот документ отражает основную операционную семантику и
правила исполнения в языке.

## 1. Модель исполнения

- **Последовательное исполнение:**
  Программные инструкции исполняются последовательно пока не
	прерываются конструкциями управления потоком исполнения
  (`если`, `пока`, `вернуть`, `выйти`, `авария`).

- **Порядок вычисления:** Выражения, аргументы функций и операнды
  вычисляются в строгом порядке *слева направо*

- **Точка входа:** Исполнение начинается именно в функции `начало`.
  Компилятор распознает только следующие сигнатуры:
  1. `функ начало: ряд[стр] аргументы -> ц32 { ... }`
  2. `функ начало: -> ц32 { ... }`
	    (Аргументы отсутствуют. Аргументы командной оболочки игнорируются).
    _(Компилятор отображает это на C++ `int main`._
    _Возвращаемое значение указывает код выхода в ОС)._

- **Паника:** Всякая неверная операция
  (напр., выход за границы, неверное приведение типа, или вызов `авария`)
	вызывает фатальную, неуловимую панику. Она выполняет именно
	C++ разматывание стека чтобы освободить память и
	вызвать сброс буфера Ввода/Вывода, затем прекращает выполнение программы.

- **Обработка ошибок:** Язык не предоставляет такие механизмы,
  как разматывание стека или исключения. Ошибки относятся к двум категориям:
  - **Обрабытываемые:** Функции уведомляют об ошибке через возвращаемое
	  значение (напр., `лог` для "успех/провал",
		`стр?` для операций, которые могут провалиться).
    Вызывающий отвечает за проверку результата.
  - **Фатальные:** Такие ошибки возникают либо из-за
	  намеренного вызова `авария(сообщение)`, либо из-за
		автоматических паник во время исполнения
    (выход за границы, искажающее приведение типа), и
	  прекращает выполнение программы.

- **Обязательные к проверке значения (Модификатор `!`):** Для обеспечения
  надежной проверки ошибок возвращаемый тип функции может иметь префикс `!`
  (напр., `-> !лог`) _(см. `grammar.md` параграф 2.4 "Объявления"_
  _пункт "<ret_type>")_. Если вызывается функция с типом возврата `!`, и ее
  результат не присваивается переменной и не используется напрямую в
  условии/выражении (т.е. значение игнорируется), семантический анализатор
  компилятора вызывает **строгую ошибку компиляции**.

## 2. Память и присваивание

Чтобы сохранить простоту компилятора и устранить неопределенность,
расположение типа в физической памяти неизменно закреплено за
его определяющим ключевым словом.
Язык опирается на правило единственности владения памятью.

- **Правило присваивания:** Присваивание (`=`) приносит владение посредством
  неукоснительной глубокой копии, сохраняя логическую независимость. Компилятор
	оптимизирует это до перемещения (`move`) в случае временных объектов
	(напр. возвращаемое значение функции).

- **Значения и единственность владения:**
  - **Типы размещенные в стеке (Передаются строго по копии значения):**
    Эти типы данных располагаются только в стеке.
    _(см. `types.md` параграф 2_
    _"Типы данных, размещенные в стеке (Передача по значению)")_
  - **Типы размещенные в куче**
	**(Передаются по копированной единственности владения):**
    Эти типы данных располагаются исключительно в куче.
	  При присваивании происходит новое выделение памяти в куче,
	  копирующее содержимое полностью.
    _(см. `types.md` параграф 3_
    _"Типы данных, размещенные в куче (Единственность владения/Заимствование)")_

- **Гарантия инициализации переменной:** Ввиду ограничений `grammar.md`
  _(см. `grammar.md` параграф 2.4 "Объявления" пункт "<var_decl>")_,
  неинициализированные переменные не могут существовать. Каждое объявление
	должно содержать явное инициализирующее присваивание,
	обеспечивая определенность состояний памяти.

- **Вычисление оператора распространения (`...`):**
	Когда используется синтаксис операции распространения
  _(см. `grammar.md`:_
  _параграф 2.4 "Объявления" пункт "<var_decl>"_,
  _параграф 2.6 "Выражения" пункты:_
    _"<spread_init>", "<struct_init>")_
  (напр., `<expr>...` при заполнении статического массива литералом),
  `<expr>` вычисляется **один раз**.
  Затем итоговое значение копируется насквозь по всему массиву.

- **Ограничения:**
  - _Нет циклически связных структур:_ Строгая единственность владения
    математически делает невозможным создание таких структур.
  - _Ограничение аппаратуры при глубокой рекурсии:_ Удаление или присваивание
	  глубоко вложенных объектов может вызвать переполнение стека в ОС.

## 3. Заимствование и изменчивость

Переменные не могут хранить ссылки. Разделение состояния без
копирования возможно только через параметры функций:

- **Передача по значению** (Примитивы без `изм`):
  Клонируются при входе в функцию.

- **Заимствование только для чтения** (Сложные типы данных без `изм`):
  Передаются как неизменные ссылки. Изменения со стороны функции строго
  запрещены компилятором.

- **Изменяемое заимствование** (`изм`):
  Передается как прямая ссылка. Соглашение на изменение должно быть явным
  с обеих сторон: сигнатура вызываемой функции должна объявлять параметр с
  `изм`, а вызывающая сторона должна ставить префикс `изм` перед
  передаваемым аргументом во время вызова. Функция взаимодействует напрямую с
  блоком памяти вызывающей стороны.

- **Неизменность (ограничение `пост`):**
  Объявление переменной с `пост` помечает ее как строго **глубоко** неизменяемую
  на протяжении всего времени жизни в ее области видимости. Запрещено применять
  модификатор `изм` к переменной `пост`, а изменение любых вложенных полей,
  элементов массива или символов строки, которыми она владеет, является
  строгой ошибкой компиляции.

- **Правила изменяемого псевдонимирования (Mutable Aliasing):** Передача
  идентичной переменной в параметр `изм` и в любой другой параметр в том же вызове
  является строгой ошибкой компиляции. Анализатор AST использует сопоставление
  структурных путей (Structural Path Matching) для обеспечения
  исключительности:
  - _Раздельные поля структуры:_ Передача различных полей
    (напр., `изм узел.левый`, `изм узел.правый`) гарантированно разделена и
    **разрешена**.
  - _Динамические индексы:_ Поскольку индексы во время исполнения не могут быть
    проверены, вызов `моя_функ(изм м[i], изм м[j])` строго запрещен.
    **Исключение:** Встроенная функция `обмен(изм м[i], изм м[j])` выполняет
    безопасную динамическую проверку и разрешена для безопасного обмена.

## 4. Посвященное системе типов

### А. Преобразования типов (Приведение):

(_Сюда ссылаются_: `types.md` параграф 6
"Синтаксис и правила преобразования типов (Приведение)")

- **Неявные преобразования:** Строго запрещены во избежание непреднамеренного
  числового усечения, за одним исключением для опциональных типов (см. ниже).

- **Явные приведения:** Выполняются с использованием оператора
  `как<тип>(выражение)`.
  _(см. `grammar.md` параграф 2.6 "Выражения" пункт "<unary>")_
  Ограничены взаимными преобразованиями между встроенными числовыми форматами,
  логическими типами и извлечением (unwrap) опциональных типов.

- **Правило паники:** Если сужающее приведение числового типа вычисляется в
  значение, превышающее вместимость целевого типа (например, усечение `ц64` 300
  в `ц8`), исполнение останавливается посредством паники.

- **Преобразования опциональных типов:**
  - **Расширение (Безопасное, Неявное):** Неопциональное значение типа `T` может
    быть неявно преобразовано и присвоено опциональному типу `T?`.
    Эта операция всегда безопасна.
  - **Сужение (Извлечение, Явное):** Преобразование опционального типа `T?` в
    неопциональный `T` потенциально небезопасно и строго запрещено неявно.
    Оно должно выполняться явно с использованием явного приведения.
    Это приведение выполняет проверку во время исполнения: если опционал
    содержит значение, оно извлекается и возвращается; если опционал содержит
    `ничто`, приведение вызывает фатальную панику. Такое поведение требует от
    программиста явной обработки случая `ничто` перед приведением.
- Встроенные функции `в_стр` и `из_стр` должны использоваться для преобразования
  в/из строк. _(см. `builtins.md` параграф 3 "Операции со строками")_

### Б. Void, Nullability и двойственная природа `ничто`:

(_Сюда ссылаются_: параграф 6 "Control Flow",
`types.md` параграф 4 "Void, Nullability и Опциональные типы"
пункт "Определение и ограничения")

- Переменные по умолчанию строго не могут быть null. Только типы данных,
  размещенные в куче и явно обозначенные как опциональные (с помощью
  добавления `?` (напр., `МояCтруктура?`)), могут хранить `ничто` (состояние
  null). Примитивы в стеке этого не могут.
  _(см. `grammar.md` параграф 2.3 "Типы" пункт "<type>")_
  Литерал `ничто` является допустимым
  выражением инициализации только для опциональных типов размещенных в куче.
  Присваивание его в других местах является ошибкой компиляции.

- `ничто` используется одновременно как возвращаемый тип `void` для функций,
  которые не возвращают значение, и как значение состояния `null` для явно
  указанных опциональных типов, размещенных в куче.

- Литерал `ничто` не имеет собственного неотъемлемого типа (inherent type).
  Он не может быть использован напрямую в контекстах, требующих неявного вывода
  типа (напр., как аргумент значения во встроенной функции `заполнить`). Для
  представления динамического массива пустых опционалов необходимо использовать
  явное приведение типа для определения базового типа:
  `заполнить(как<стр?>(ничто), 10)`.

### В. Вывод типа литерала

Числовые литералы (напр., `10`, `3.14`) не обладают присущим им строгим типом
во время первоначального синтаксического анализа.

- **Контекстная типизация:** Когда числовой литерал используется в контексте со
  строго ожидаемым типом (напр., при присваивании объявленной переменной или
  при передаче в типизированный параметр функции), литерал неявно принимает
  целевой тип. Компилятор статически проверяет, что значение литерала физически
  помещается в границах целевого типа; если происходит переполнение,
  выбрасывается ошибка компиляции.

- **Типизация по умолчанию:** Когда числовой литерал используется в нестрогом
  выражении без явного ожидаемого типа (напр., передан в перегруженную
  встроенную функцию наподобие `в_стр(5)`, или использован в самостоятельной
  проверке равенства наподобие `5 == ничто`), целочисленные литералы строго
  по умолчанию получают наименьший стандартный знаковый целочисленный тип,
  способный физически представлять это значение, с минимальным порогом `ц32`
  (т.е. по умолчанию используется `ц32`, или `ц64`, если значение превышает
  вместимость `ц32`). Литералы с плавающей точкой по умолчанию используют
  `пт32`, или `пт64`, если значение превышает вместимость `пт32`.

## 5. Выражения и операции

### А. Поддерживаемые операции:

- **Арифметические (определены для числовых типов):**
  `+` (сложение), `-` (вычитание/унарное отрицание),
  `*` (умножение), `/` (деление), `%` (остаток от деления).
  _Перегрузки:_
  `+` объединяет строки (конкатенация). `*` повторяет строку целое число раз.

- **Побитовые (только для целочисленных примитивов):**
  `&` (AND), `|` (OR), `~` (Инверсия), `^` (XOR),
  `<<` (Сдвиг влево), `>>` (Сдвиг вправо).

- **Операции отношения (определены для примитивов):**
(_Сюда ссылаются_:
`types.md` параграф 4 "Void, Nullability и Опциональные типы"
пункт "Сравнение с Null",
`language_specification.md` параграф 8 "Type Casting")
  `==`, `!=`, `<`, `>`, `<=`, `>=`.
  _Перегрузки:_ `==` и `!=`:
  1) Проверка равенства строк `стр` (сравнивает базовое содержимое UTF-8).
  2) Проверка состояния null: сравнение любого опционала с литералом `ничто`.
  3) Сравнение опционального типа с таким же неопциональным запрещено.
  _Явное исключение:_ `стр?` может быть напрямую сравнена с `стр`.
  Сравнение симметрично.
  Если опционал содержит `ничто`, `==` безопасно вычисляется в `ложь`.

- **Логические:** `и` (AND), `или` (OR), `не` (NOT).

- **Операторы доступа:**
  `.` (Доступ к полю), `[]` (Индексация или срез строки),
  `\` (Разрешение области видимости / пространства имен).

- **Присваивания:** `=`, `+=`, `-=`, `*=`, `/=`.

- **Группировка:** `()`

### Б. Семантика выражений и операторов

- **Приоритет операторов:**
  _(см. `grammar.md` параграф 2.7 "Приоритеты и ассоциативность операторов")_

- **Укороченное вычисление (Short-Circuiting):** Логические `и` (AND) и
  `или` (OR) вычисляются по укороченной схеме. Если левая часть удовлетворяет
  условию, правая часть **не вычисляется**. Любые вызовы функций или побочные
  эффекты в правой части безопасно пропускаются.

- **Прозрачный доступ к полям структуры (`.`):**
(_Сюда ссылаются_: `codogen.md` параграф 3
"Семантика памяти и сопоставления параметров"
пункт "Трансляция доступа к полям")
  Оператор точка действует как универсальный метод доступа к полям структуры.
  Пользователь применяет идентичный синтаксис `.` независимо от того, является
  ли структура значением, привязанным к стеку (`стат_структ`), или умным
  указателем, привязанным к куче (`структ`).
  
- **Индексация и срезы (`[i]` и `[начало:конец]`):**
  - Индексы должны строго вычисляться в беззнаковое целое число
    (`ц8б`, `ц16б`, `ц32б`, `ц64б`).
  - **Индексация массивов:** Динамические и статические массивы
    поддерживают только доступ по индексу к одному элементу (`[i]`). Выполнение
    среза массива является ошибкой компиляции.
  - **Срезы строк:** Выполнение среза (`[начало:конец]`) доступно
    только для `стр`. Оно вычисляется в
    **совершенно новую, глубоко скопированную строку**, содержащую символы
    начиная с индекса `начало` вплоть до, но не включая, индекс `конец`.
    Если `начало >= конец`, возвращается пустая строка (`""`).
    Если любой предоставленный индекс выходит за границы, операция вызывает
    фатальную панику.

- **Семантика индексации строк UTF-8:**
  Доступ к элементам строки через `[i]` использует **индексацию символов UTF-8**
  (а не смещения байтов) и возвращает односимвольную `стр`, доступную только для
  чтения. Поскольку `стр` под капотом отображается на массив байтов,
  извлечение _i-го_ UTF-8 символа является **операцией на O(N)**.
  *Предупреждение*:
  Последовательный перебор строки с использованием цикла по индексу приводит к
  алгоритму O(N^2). Для последовательной обработки символов явно преобразуйте
  строку в `ряд[ц32]`, используя встроенную функцию `в_ряд_символов`.
  _(см. `builtins.md` параграф 3 "Операции со строками"_
  _пункт "Преобразование строки в массив UTF-32")_
  Присваивание элементу строки по индексу является ошибкой компиляции.

- **Ограничения доступа к опционалам:** Применение операторов доступа к полям
  структуры (`.`) или индекса/среза (`[]`) напрямую к переменной опционального
  типа (`?`) является **строгой ошибкой компиляции**. Чтобы получить доступ к
  базовому значению, программист должен сначала явно и безопасно извлечь
  значение. Обычно это делается путем использования явного приведения типа,
  которое выполняет "извлечение-или-панику", тем самым заставляя программиста
  осознанно признавать возможность ошибки во время исполнения, если значение
  равно `ничто`.
- **Присваивания как инструкции:** Операторы присваивания (`=`, `+=` и т.д.)
  явно синтаксически анализируются как **инструкции**, а не выражения. Это
  естественно предотвращает их использование внутри условных операторов
  (напр., `если (а = б)` вызывает ошибку компиляции).

## 6. Управление потоком исполнения

- `если`: Условно вычисляет `лог` (булево) выражение. Анализатор компилятора
  строго следит за тем, чтобы условие вычислялось _исключительно_ в `лог`.
  Неявное вычисление числовых типов (подобно `if (1)` в C) является строгой
  ошибкой компиляции, предотвращая проникновение неявных преобразований C++
  через транслятор.

- `пока`: Выполняет блок исполнения в цикле, пока условие истинно (`истина`).

- `прекратить` (break): Немедленно заставляет исполнение выйти из наиболее
  глубоко вложенной структуры цикла `пока`.

- `продолжить` (continue): Немедленно останавливает выполнение текущего блока
  итерации и переходит обратно к вычислению условия `пока` для следующей
  возможной итерации.

- **Полнота возврата:** Каждая функция с типом возвращаемого значения, отличным
  от `ничто`, обязана гарантировать возврат значения. Компилятор обеспечивает это
  с помощью строгой однопроходной **консервативной структурной проверки**.
  Функция структурно проходит проверку тогда и только тогда, когда выполняется
  одно из условий:
  1. Последней инструкцией верхнего уровня функции является именно
     инструкция `вернуть ...;` **ИЛИ** самостоятельный вызов
     `авария(...);` или `выйти(...)`.
  2. Последняя инструкция является блоком `если ... иначе ...`, где _и_ `если`,
     _и_ `иначе` ветви структурно заканчиваются на `вернуть`, `авария`
     или `выйти`.
     _ПРИМЕЧАНИЕ: Семантический анализатор AST должен в явном виде_
     _рассматривать самостоятельный вызов `авария` или `выйти` как_
     _конечный узел потока исполнения конструкции (эквивалентный `вернуть`),_
     _а не как стандартный вызов функции, чтобы удовлетворить эту проверку._
     _Любой другой структурный поток, даже если он математически гарантированно_
     _содержит бесконечный цикл или завершится оператором вернуть, считается_
     _неполным и вызывает ошибку компиляции._
- **`ничто` в инструкциях возврата (Двойственная семантика):** `вернуть ничто;`
  ведет себя контекстно, основываясь на объявленном возвращаемом типе функции.
  _(Согласно параграфу 4.Б "Void, Nullability и двойственная природа `ничто`")_

## 7. Разрешение имен, области видимости и модули

- **Лексическая область видимости блоков:** Переменные, определенные внутри
  блока `{}`, уничтожаются при завершении блока.

- **Ограничения затенения (Shadowing):** Внутренние области видимости могут
  перекрывать (затенять) переменные с идентичными именами из внешних областей.
  Однако, повторное объявление переменной с таким же именем в _той же самой_
  области видимости является ошибкой компиляции.

- **Псевдонимы типов (Type Aliasing):**

(_Сюда ссылаются_: `types.md` параграф 7
"Псевдонимы типов (Type Aliasing)")

- - Ключевое слово `употреб` создает **прозрачный псевдоним**
    (эквивалентно C++ `using`).
- _Пример:_ `употреб ц32 Возраст;`
  _(см. `grammar.md` параграф 2.2 "Модули, пространства имен и псевдонимы"_
  _пункт "<using_stmt>")_
- Будучи объявленными, `Возраст` и `ц32` на 100% взаимозаменяемы в сигнатурах
  функций, присвоениях и выражениях. Это не создает новый, строгий и
  изолированный тип.

- **Жизненный цикл изменчивости переменных:** Переменные, объявленные как
  `пост`, являются полностью неизменяемыми на протяжении всего их жизненного
  цикла. Применение `изм` к переменной `пост` — ошибка компиляции.

- **Однопроходное объявление:** Компилятор выполняет синтаксический
  анализ сверху-вниз. Использование _любого_ идентификатора (переменной,
  функции, структуры) до его лексического объявления строго запрещено.
  Вызываемые функции или структуры, к которым происходит обращение до их
  определения, должны иметь предварительные объявления (прототипы).

- **Глобальные переменные:** Переменные, объявленные на верхнем уровне файла,
  имеют время жизни (lifetime) в рамках всей программы. Они доступны из всех
  функций, определенных после их объявления.

- **Пространства имен (`область`):** Инкапсулируют области видимости. Доступ
  к ним возможен через явный префикс. Пустые области запрещены.
  - _Импорт:_ `употреб МояОбласть\*;`
  (Подстановочный знак `\*` применяется исключительно к контейнерам `область`).
   _(Синтаксис см. `grammar.md` параграф 2.2_
   _"Модули, пространства имен и псевдонимы" пункт "<using_stmt>")_

- **Пустые структуры:** Объявление `структ` или `стат_структ` с нулевым
  количеством полей является ошибкой компиляции.

- **Включение модулей:** Анализируется строгим однопроходным методом.
  - `включить "файл.ru";` — Раскрывает все объявления верхнего уровня начиная
    с места вызова вниз.
  - `из "файл.ru" включить имя1, имя2;` — Раскрывает конкретно указанные
    идентификаторы. _(Синтаксис см. `grammar.md` параграф 2.2_
  _"Модули, пространства имен и псевдонимы" пункт "<include_stmt>")_
  - Инструкция вызова `включить` внутри пространства имен прикрепляет включенные
    идентификаторы к этому текущему пространству имен (области).
  - **Идемпотентное и аддитивное включение файлов:**
    - Включаемые файлы анализируются в AST только один раз. Идентификаторы
      добавляются в область видимости последовательно и идемпотентно.
    - Циклические зависимости включений должны быть разорваны вручную с помощью
      прототипов структур, объявленных перед инструкцией `включить`.

## 8. Автомат состояний Ввода/Вывода

(_Сюда ссылаются_: `builtins.md` параграф 1
"Ввод/Вывод и взаимодействие с файлами"
пункт "Контроль целей (Перенаправление)")

- **Парадигма активного автомата состояний:** Программа поддерживает две
  глобальные "активные цели": одну для чтения и одну для записи. По умолчанию,
  они отображаются на стандартные потоки консоли (`stdin`, `stdout`). Все
  инструкции ввода/вывода динамически направляются к текущим активным целям и
  взаимодействуют с ними.
- Установление новой активной цели осуществляется посредством
  специальных встроенных функций _(см. `builtins.md` параграф 1_
  _"Ввод/Вывод и взаимодействие с файлами")_, причем оно автоматически и
  безопасно сбрасывает данные буфера (flush) и закрывает предыдущую файловую
  цель, полностью исключая ошибки или утечки при ручном управлении
  дескрипторами.
- **Правило отступления (Fallback Rule):** Если какая-либо функция
  перенаправления завершается неудачно (напр., файл не найден, отказано в
  доступе) и возвращает `ложь`, активная в данный момент цель ввода/вывода
  _не_ закрывается и остается полностью активной.
