# Генерация кода (Транспиляция в C++)

Этот документ в общих чертах описывает, как AST языка структурно отображается
на целевую фазу генерации C++.

_ПРИМЕЧАНИЕ: Это предварительная спецификация MVP, отражающая_
_ключевые аспекты реализации._

## 1. Архитектура целевого бэкенда

- **Целевой язык:** Строго ISO C++17 (или C++20).
- **Конвейер компиляции:** Синтаксический анализатор языка внутренне
  генерирует единую консолидированную единицу трансляции `.cpp`. Этот
  полученный код затем передается стандартному C++ компилятору
  (такому как GCC или Clang) для создания высокооптимизированного нативного
  исполняемого бинарного файла.

## 2. Основные правила сопоставления типов

- `цN` -> `intN_t` (напр., `ц32` -> `int32_t` через `<cstdint>`)
- `цNб` -> `uintN_t`
- `пт32` -> `float`, `пт64` -> `double`
- `лог` -> `bool`
- `стр` -> `std::string`
- `ряд[T]` -> `std::vector<T>`
  - _Исключение:_ `ряд[лог]` ОБЯЗАН отображаться на `std::vector<uint8_t>`
    (или `std::vector<char>`). Это предотвращает применение C++ его
    упакованного по битам прокси `std::vector<bool>`, гарантируя, что элементы
    можно безопасно передавать по изменяемой ссылке (`изм`).
- `стат_ряд[T, N]` -> `std::array<T, N>`
  - Если `N` является заданным пользователем идентификатором размера в
    параметре функции (напр., `стат_ряд[ц32, размер]`), каждое уникальное имя
    генерирует отдельный параметр `template<size_t размер>` для сгенерированной
    C++ функции. Этот идентификатор доступен внутри тела функции как
    `const uint32_t`.
  - _Распространение цепочки шаблонов:_ Если шаблонная функция `ф1` вызывает
    другую шаблонную функцию `ф2`, передавая свой собственный именованный
    статический массив, C++ транспилятор автоматически распространяет и
    разрешает аргументы шаблона `size_t`.
    Эта цепочка инстанцирования абсолютно невидима для пользователя языка.
  - _ПРИМЕЧАНИЕ для прототипов:_ Если такая функция объявлена предварительно
    (прототип), транспилятор обязан также генерировать прототип как
    шаблонное предварительное объявление (forward declaration) в C++.
- `структ X` -> `std::unique_ptr<X>`
- `стат_структ X` -> `struct X { ... }`
- **Опциональные типы (суффикс `?`):**
  - `структ X?` -> `std::unique_ptr<X>`
    (Поскольку C++ `unique_ptr` изначально поддерживает состояние null).
    Литерал `ничто` отображается на `nullptr`.
  - `стр?` -> `std::optional<std::string>` (через `<optional>` в C++17).
    Литерал `ничто` отображается на `std::nullopt`.
  - `ряд[T]?` -> `std::optional<std::vector<T>>`.
    Литерал `ничто` отображается на `std::nullopt`.

## 3. Семантика памяти и сопоставления параметров

- **Присваивания и шаблонный код (Boilerplate):** Генерация присваиваний
  безукоризненно нативно обеспечивает правило "Глубокой копии". Поскольку
  стандарт C++ математически определяет `std::unique_ptr` строго как
  присваиваемый только перемещением (move-only), это естественно приводит к
  автоматическому удалению конструкторов копирования всей структуры.

- **Трансляция доступа к полям:**
  Поскольку язык использует универсальный оператор `.` для любого доступа
  к полям _(см. `semantics.md` параграф 5.Б_
  _"Семантика выражений и операторов" пункт_
  _"Прозрачный доступ к полям структуры (`.`)")_,
  транспилятор C++ должен структурно разрешать базовый тип.
  Если целью является `стат_структ`, он применяет стандартную точку C++ (`.`).
  Если целью является `структ` (которая отображается на указательподобный
  `std::unique_ptr`), он динамически подставляет оператор стрелочки C++ (`->`).

- **Договоренность генерации кода:** Если любая проанализированная структура
  (`структ` или `стат_структ`) содержит поля размещенные в куче,
  AST транспилятор обязан создать пользовательские конструкторы
  глубокого копирования и логику `operator=`. По умолчанию, копирование
  запускает вычисление дерева условий (напр.
  `this->узел = other.узел ? std::make_unique<Узел>(*other.узел) : nullptr;`)
  для полей `unique_ptr`, и безопасно копирует размеры для коллекций
  примитивов.

- **Универсальное смягчение последствий глубокого копирования:**
  C++ удаляет операторы присваивания для полей `unique_ptr`.
  Чтобы соблюсти правило глубокой копии среди вложенных коллекций,
  транспилятор отображает `a = b` на глобально перегруженный набор
  шаблонов `_копия(T)`:

  ```cpp
  template<typename T> T _копия(const T& val) { return val; } // Primitives

  template<typename T> std::optional<T> _копия(const std::optional<T>& val) {
      return val ? std::make_optional(_копия(*val)) : std::nullopt;
  }

  template<typename T>
  std::unique_ptr<T> _копия(const std::unique_ptr<T>& val) {
      return val ? std::make_unique<T>(_копия(*val)) : nullptr;
  }

  template<typename T> std::vector<T> _копия(const std::vector<T>& val) {
      std::vector<T> res; res.reserve(val.size());
      for(const auto& item : val) res.push_back(_копия(item));
      return res;
  }
  ```

  При разборе выражения `a = b;`, включающего любой тип размещенный в куче,
  транспилятор повсеместно перехватывает инструкцию и подставляет
  `a = _копия(b);` вместо нее.

- **R-Values и семантика перемещения:** Если правая часть присваивания
  является временным значением (напр., прямым результатом вызова
  функции `а = моя_функ()`), транспилятор явно генерирует `std::move(...)`. Это
  позволяет избежать лишних выделений памяти и удовлетворяет правилам
  маршрутизации `unique_ptr` в C++ для возврата из функций, не нарушая
  иллюзию пользователя языка касательно "всегда глубокой копии" для переменных.

- **Инстанцирование и литералы:**
  - Во время транспиляции литерала `структ`
    (который отображается на размещенный в куче `std::unique_ptr`),
    транспилятор генерирует `std::make_unique<ИмяСтрукта>(...)`.
    Например, `Узел{1, 2}` становится `std::make_unique<Узел>(1, 2)`.
  - Литералы `стат_структ` отображаются на стандартную агрегатную инициализацию
    C++.

- **Интерпретация передачи параметров:**
  - Примитивы естественно транспилируются как передача по значению.
  - Сложные типы (заимствование только для чтения) естественно передаются
    как `const T&`.
  - Изменяемые переменные (`изм`) транслируются исключительно в изменяемые
    ссылки `T&`.

- **Отображение глубокой неизменности (`пост`):**
  Переменные объявленные как `пост` отображаются на `const` в C++.
  Однако, поскольку `const std::unique_ptr<T>` нативно предотвращает только
  переназначение самого указателя (а не изменение внутренних полей),
  семантический анализатор фронтенда строго обязуется перехватывать и
  блокировать любые изменения полей, индексов или использование сеттеров над
  переменными `пост структ`.
  Глубокая неизменность математически гарантируется до того, как код C++
  будет сгенерирован.

- **Строгая проверка границ (Договоренность генерации кода):**
  Стандартные методы C++ `std::vector::operator[]` и `std::string::operator[]`
  не выполняют проверку выхода за границы в режиме release.
  Для удовлетворения требований языка о вызове фатальной паники при доступе
  за пределами массива, транспилятору строго предписано отображать все
  индексации массивов на безопасные функции доступа.
  - **Массивы:**
    Все доступы по индексу `[i]` в AST должны отображаться на метод `.at(i)`
    из C++.
  - **Строки (Встроенные функции UTF-8):** Поскольку `std::string::at()`
    работает с байтами, а не с символами UTF-8, индексы строк `стр[i]`
    **не должны** отображаться на метод `.at()`. Вместо этого, транспилятор
    обязан создать и использовать глобальную внутреннюю C++ функцию
    `_индекс_стр(const std::string&, uint32_t)`, которая производит
    итерацию по последовательности UTF-8, возвращая новую `std::string`,
    содержащую одиночный символ. Если запрашиваемый индекс превышает
    количество символов UTF-8, эта встроенная функция должна вручную выбросить
    заложенное исключение паники
    (`throw _ВнутренняяАвария("Индекс вне границ");`).
  - Для обработки индексации примитивных массивов сгенерированная
    обертка запуска `main` также обязана перехватывать `std::out_of_range`
    (выбрасываемое методом `.at(i)` в C++) наравне с пользовательским
    исключением `_ВнутренняяАвария`, перенаправляя оба варианта к одинаковой
    схеме вывода логов ошибок через `stderr` и завершения процесса с
    ненулевым кодом выхода.

- **Явное извлечение опциональных типов (`как<T>`):**
  Поскольку `структ Узел` и `структ Узел?` оба транслируются нативно в
  абсолютно одинаковый тип C++ (`std::unique_ptr<Узел>`), применение
  приведения `как<Узел>(опт_узел)` механически не меняет систему типов
  со стороны C++.
  Чтобы позволить извлеченным переменным корректно передаваться в
  параметры `изм` без компиляции промежуточных R-значений, транспилятор
  обязан транслировать `как<T>` во встраиваемую (inline) проверку через
  тернарный оператор, возвращающую ссылку:
  `(опт_узел ? *опт_узел : (_авария_внутр(), *опт_узел))`
  (Где `_авария_внутр()` вызывает фатальную аварийную остановку).

- **Отображение Паники/Аварийного завершения:** Любые фатальные паники во время
  исполнения (включая неверный индекс, неудачные сужающие (narrowing)
  приведения, извлечение (unwrap) `ничто` и явные вызовы функции `авария`)
  должны отображаться на выбрасывание выделенного класса внутреннего
  вспомогательного исключения C++
  (напр., `throw _ВнутренняяАвария("сообщение");`).
  Это гарантирует правильное разматывание стека в стиле C++, позволяя
  деструкторам RAII автоматически очистить выделения памяти в куче
  (`std::unique_ptr`) и сбросить буфер активных файловых потоков Ввода/Вывода.
  Сгенерированная функция `main` должна обертывать вызов функции `начало`
  в блок `try { ... } catch (const _ВнутренняяАвария& e) { ... }`.
  При перехвате этого исключения, обертка выводит сообщение об ошибке на
  `stderr` и безопасно завершает процесс с ненулевым возвращенным
  кодом выхода в ОС.
  Язык намеренно не предоставляет пльзователю синтаксис для
  перехвата исключений.

- **Отображение обязательной проверки (Модификатор `!`):** Если разобранное
  объявление функции имеет модификатор типа возврата `!`, транспилятор обязан
  сгенерировать атрибут C++17 `[[nodiscard]]` точно как для сгенерированного
  прототипа функции C++, так и для её определения
  (напр., `[[nodiscard]] bool my_func();`).

## 4. Стратегия генерации компилятором

- Поскольку язык строго требует соблюдения однопроходного синтаксического
  анализа в стиле C, запрещающего использование необъявленных объектов,
  транспилятор естественно воспроизводит это, логически отображая явно
  сформулированные пользователем прототипы (`функ ... ;`, `структ ... ;`) на
  стандартные предварительные объявления C++ (forward declarations).

- Нативная точка входа в языке `начало(...)` точно оборачивает и отображается
  на классическое C++ представление `int main(int argc, char** argv)`.
  Сгенерированная функция `main` включает пролог, который преобразует
  `argc`/`argv` в `std::vector<std::string>` перед передачей в
  пользовательскую функцию `начало` как параметр `ряд[стр] аргументы`.

- **Разделение жизненного цикла структур:**
  **(Исключение неопределенных типов (Incomplete Type)):**
  Поскольку язык поддерживает взаимные ссылки структур друг на друга,
  а `std::unique_ptr` требует полностью определенный тип для создания
  деструкторов и логики глубокого копирования, наивная генерация C++
  сверху-вниз потерпит неудачу из-за нативных ошибок "incomplete type" при
  компиляции. Транспилятор строго обязан выдавать C++ код для структур в
  три отдельные фазы:
  1.  **Предварительные объявления:** Генерировать `struct X;` для всех
      синтаксически разобранных структур.
  2.  **Структурные схемы (Layouts):** Генерировать тела структур с их полями.
      Деструкторы и операторы присваивания-копирования обязательно должны
      быть объявлены, но _не_ реализованы на месте (inline).
  3.  **Реализации жизненного цикла:** Генерировать реализации деструкторов
      (`X::~X() = default;`) и шаблонных специализаций `_копия` _после_ того,
      как все структуры были полностью определены.

## 5. Внутренняя архитектура глобальных состояний Ввода/Вывода

- Декларативный дизайн "Активного автомата состояний" для функций
  ввода/вывода элегантно переводится в скрытые стандартные глобальные
  переменные: один объект `std::ifstream active_file_in;` и
  `std::ofstream active_file_out;`.
- Легковесные указатели-трекеры активного режима по умолчанию надежно
  указывают на объекты `std::cin` и `std::cout`, естественно меняя свои
  внутренние состояния при вызове функций целевого
  перенаправления ввода или вывода.
